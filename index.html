Chromium DWM/MPO rendering bug — minimal repro

Reproduces the rendering glitch affecting Chromium-based apps on Windows 11 24H2 / 25H2 with discrete GPUs. Tracked at Chromium 429859152 and electron/electron#46206.

How to reproduce

  1. Watch the FRAME counter and the colored canary stripe on the right edge.
  2. Press Alt+Tab to a fullscreen game/video, or click Fullscreen then Esc.
  3. Switch back. Repeat 2–3 times if needed.

Bug present: the stripe stops scrolling, or the counter freezes, while the rest of the page (cursor, buttons) still works. long pauses increments. Click the page → it resumes. The bug is in the OS compositor (DWM); the renderer is fine.

No bug: the stripe never stops, counter increments continuously, and long pauses stays at 0 (or only ticks when visibility actually changed).

What the metrics show

FRAME / rAF fps — animation frames the page rendered. If these stop incrementing while you can still see and interact with the page, the compositor is frozen.

video fps / video frames — frames the browser actually presented to the screen, via the requestVideoFrameCallback API. This number lagging behind elapsed × 24 is direct evidence the compositor dropped your frames.

video dropped — count from getVideoPlaybackQuality().droppedVideoFrames, the browser's own dropped-frame counter.

long pauses — number of times requestAnimationFrame went silent for over 200ms while the page was visible and focused.

Workarounds

      
Per-app: chrome://flags/#use-angle → set to D3D9 or OpenGL.
         (Note: loses HEVC hardware decode in some configurations.)

      
Win 11 24H2 system-wide: HKLM\SYSTEM\CurrentControlSet\Control\GraphicsDrivers OverlayMinFPS=0, restart.

      
Win 11 25H2 system-wide: HKLM\SYSTEM\CurrentControlSet\Control\GraphicsDrivers DisableOverlays=1, restart.
